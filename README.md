# 프로젝트 운영 초안

## 문서의 목적

이 문서는 우리 스터디 프로젝트를 어떤 기준과 방향으로 운영할지 정리한 초안입니다.  
일정표를 세우기 위한 문서라기보다, 우리가 어떤 기준으로 프로젝트를 진행할지 먼저 맞춰보기 위한 안내문에 가깝습니다.

이번 프로젝트에서 가장 중요하게 보고 싶은 것은 화려한 결과물 자체가 아니라,  
**협업 구조를 갖춘 상태에서 제한된 구현을 끝까지 해보는 경험**입니다.

즉, 이번 스터디는 “큰 서비스를 만드는 것”보다도 아래와 같은 경험을 남기는 데 더 의미를 두려고 합니다.

- GitHub 기반 협업 흐름을 실제로 경험하는 것
- 기능을 나누고 역할을 분담해보는 것
- 구현 전에 구조와 책임을 먼저 고민해보는 것
- README와 문서로 작업 이유와 과정을 남기는 것
- 범위를 통제한 결과물 1개를 끝까지 완성하는 것
- Java 또는 JS를 실제 프로젝트 맥락에서 적용해보는 것

이 문서는 그 방향을 팀원들과 공유하고, 이후 회의에서 함께 조정해가기 위한 출발점입니다.

---

## 1. 이번 프로젝트를 바라보는 기본 전제

현재 팀은 4명으로 운영되고, 팀원 모두 Java, Front(Web), Algorithm에 대해 최소한의 기초는 갖추고 있다고 봅니다.  
다만 백엔드는 아직 배우지 않았고, 프로젝트 경험도 많지 않습니다.

그래서 이번 프로젝트는 처음부터 완성도 높은 서비스나 풀스택 결과물을 목표로 잡기보다,  
**현재 수준에서 충분히 이해하고 설명할 수 있는 범위를 정하고 그 안에서 협업과 설계를 경험하는 방향**으로 가는 것이 맞다고 생각합니다.

프로젝트 기간은 약 2개월, 기본 결과물은 1개를 전제로 두겠습니다.  
주제와 언어는 회의 후 결정하되, 문서상으로는 **JS 방향과 Java 방향을 모두 열어두는 것**을 원칙으로 하겠습니다.

---

## 2. 우리가 이번 프로젝트에서 남기고 싶은 핵심

이번 프로젝트에서 결과물보다 더 중요하게 남기고 싶은 것은 다음과 같습니다.

### 2-1. 협업 경험
각자 기능을 나누어 맡고, 작업 내용을 기록하고, Git 흐름 안에서 충돌과 조율을 경험하는 것 자체가 이번 프로젝트의 핵심입니다.  
단순히 코드를 합치는 것이 아니라, **어떻게 나누고 어떻게 합칠지**를 고민해보는 경험을 만들고 싶습니다.

### 2-2. 구조 설계 경험
바로 구현부터 들어가기보다,  
어떤 기능이 필요한지, 어떤 책임을 분리해야 하는지, 어떤 파일 구조나 패키지 구조가 적절한지 먼저 정리한 뒤 구현에 들어가는 흐름을 가져가고 싶습니다.

### 2-3. 문서화 경험
README, 작업 기록, 기능 설명, 예외 케이스 정리처럼  
“우리가 왜 이렇게 만들었는지”를 남기는 작업도 이번 프로젝트의 중요한 결과물로 봅니다.

### 2-4. 제한된 범위의 완성 경험
범위를 넓히다가 흐려지는 프로젝트보다,  
조금 작더라도 기준이 분명한 결과물 하나를 끝까지 완성하는 것이 더 중요합니다.

## 3. 최우선 목표

아래 항목들은 이번 프로젝트에서 가장 분명하게 가져가야 하는 핵심 목표입니다.

### 3-1. 기능별 역할 분담
기능을 단위로 나누고, 각자 맡은 범위를 분명히 하겠습니다.  
누가 어떤 기능을 맡고 있는지, 어떤 책임을 갖는지, 공통 영역은 어디까지인지가 정리되어 있어야 합니다.

### 3-2. Git 협업 관리
저장소를 단순 공유 폴더처럼 쓰지 않고, 협업 도구로 사용하겠습니다.

최소한 아래 항목은 반드시 지키는 방향으로 가겠습니다.

- branch 분리
- commit 메시지 가독성 확보
- 병합 기준 합의
- README 및 작업 기록 유지
- 역할별 작업 흐름 공유

### 3-3. 페이지 접근 제어 (Web Project를 진행하는 경우)
학습용 수준이더라도,  
사용자가 특정 상태가 아닐 때 어떤 페이지에 접근할 수 있는지, 접근이 막힐 때 어떤 흐름으로 이동시킬지를 경험해보겠습니다.

중요한 것은 “보안적으로 완전한 인증 시스템”이 아니라,  
**상태 확인 → 접근 판단 → 안내 메시지 → 이동 처리** 흐름을 직접 설계하고 구현해보는 것입니다.

### 3-4. 예외 메시지와 이동 로직 처리
입력이 잘못되었거나, 접근 조건이 맞지 않거나, 필요한 데이터가 없을 때  
어떻게 사용자에게 알려주고 어디로 이동시킬지를 분명히 하겠습니다.

단순히 오류를 막는 것보다,  
**왜 막히는지 보여주고 다음 행동으로 연결하는 흐름**을 만드는 것이 목표입니다.

### 3-5. JSON 기반 사용자 조회
DB를 무리하게 도입하기보다,  
JSON 기반으로 사용자나 기타 도메인 데이터를 조회하고 검증하는 흐름을 경험하는 것을 우선하겠습니다.

즉, 이번 프로젝트에서는  
“데이터를 어떻게 구조화해서 읽고, 가공하고, 조건에 따라 판단하는가”를 먼저 다루겠습니다.

### 3-6. 프론트와 얇은 백엔드 사이의 요청 및 응답 흐름 만들기
가능하다면 이번 프로젝트에서 최소한의 요청-응답 흐름까지는 경험해보는 것을 목표로 하겠습니다.

다만 여기서 말하는 백엔드는 무거운 서버 구조가 아니라,  
**프론트의 입력을 받고, JSON 또는 간단한 규칙에 따라 처리한 뒤, 결과를 다시 반환하는 얇은 흐름**을 뜻합니다.

이 흐름은 서비스 완성도를 과시하기 위한 것이 아니라,  
프론트와 처리 로직이 어떻게 연결되는지 이해하기 위한 학습 목표로 보겠습니다.

## 4. 조건부 목표

아래 항목들은 되면 좋지만, 이번 프로젝트의 필수 목표로 두지는 않겠습니다.

### 4-1. JS 또는 Spring 기반의 얇은 백엔드 구현
JS 기반의 간단한 서버는 현재 팀 기준에서 비교적 현실적인 선택지일 수 있습니다.  
반면 Spring은 현재 수준에서 학습 부담이 커질 수 있으므로, 하게 되더라도 핵심 성과처럼 과장하지 않고 제한적 확장으로만 다루겠습니다.

즉, 백엔드 연결은 **핵심 목표가 아니라 선택적 확장 요소**입니다.

### 4-2. 기능에 대한 예외 처리 확장
기본적인 예외 메시지와 이동 로직은 최우선 목표에 포함되지만,  
보다 세분화된 예외 케이스까지 넓히는 것은 프로젝트 진행 여유에 따라 판단하겠습니다.

## 5. 기술 방향은 회의 후 결정하되, 지금은 두 방향을 모두 열어둡니다

현 시점에서는 특정 기술을 확정하기보다,  
우리 팀이 어떤 경험을 더 우선할지에 따라 두 방향 중 하나를 선택하는 것이 맞다고 봅니다.

### 5-1. JS 중심으로 진행하는 경우
이 방향은 **기능 흐름과 웹 체험성**을 조금 더 살리기 좋은 방향입니다.

핵심 이미지는 다음과 같습니다.

- 프론트
- JSON 기반 데이터 처리
- 얇은 서버
- 간단한 로그인 흐름
- 예외 처리
- Git 협업 구조
- 설계 문서화

이 방향에서는 아래 경험을 상대적으로 더 직접적으로 가져갈 수 있습니다.

- 페이지 이동
- 상태 확인
- session storage 활용
- 요청-응답 흐름 경험
- 이벤트 처리와 화면 기반 분업
- 공통 유틸 분리와 모듈 구조 설계

즉, 사용자가 입력하고, 상태를 확인하고, 특정 페이지 접근을 제어하고, 결과를 화면에서 확인하는 흐름을 비교적 직관적으로 다루기 좋습니다.

### 5-2. Java 중심으로 진행하는 경우
이 방향은 **도메인 설계와 로직 구조화**에 좀 더 집중하기 좋은 방향입니다.

핵심 이미지는 다음과 같습니다.

- 도메인 설계
- 데이터 가공
- 예외 처리
- 파일 입출력
- Git 협업 구조
- 최소 시연 구조

이 방향에서는 user, comment, follow 같은 데이터를  
객체와 규칙 중심으로 다루면서 구조 설계와 로직 구현에 집중할 수 있습니다.

즉, Java 방향은 아래와 같은 장점이 있습니다.

- 객체 중심 설계 경험을 살리기 좋다
- 데이터 검증과 가공 로직이 선명하다
- 패키지 구조와 책임 분리를 연습하기 좋다
- 핵심 로직을 팀이 직접 구현했다는 밀도가 높아질 수 있다

대신 화면 체험성은 상대적으로 약할 수 있으므로,  
시연 방식은 콘솔, 파일 출력, 최소 UI 보조 구조 등으로 현실적으로 잡는 것이 좋겠습니다.

### 5-3. 어떤 방향이든 공통으로 지켜야 하는 것
언어가 무엇이든, 아래는 공통 기준으로 가져가겠습니다.

- 구조를 먼저 정리하고 구현한다
- 역할 분담이 가능한 단위로 기능을 나눈다
- 공통 규칙을 문서로 남긴다
- 결과물보다 협업 과정과 설계 이유를 더 중요하게 본다
- “우리 수준에서 직접 이해하고 설명할 수 있는 범위”를 넘어서지 않는다

## 6. 운영 방식의 기본 원칙

### 6-1. 구현보다 먼저 기능 분해
바로 코드부터 치기보다,  
먼저 “무슨 기능이 필요한지”, “이 기능은 어떤 책임을 가지는지”, “어디까지를 이번 프로젝트 범위로 볼지”부터 정하겠습니다.

### 6-2. 역할은 되도록 겹치지 않게, 그러나 완전히 분리하지도 않게
각자 맡은 기능은 분명히 하되,  
전체 흐름을 아무도 모르는 상태가 되지 않도록 중간 공유와 구조 설명을 병행하겠습니다.

즉, 각자 담당은 있지만 프로젝트 전체에 대한 공통 이해도도 함께 가져가야 합니다.

### 6-3. 문서와 코드가 따로 놀지 않게 하기
README, 기능 정리 문서, 규칙 문서는 형식만 남기는 문서가 아니라  
실제 구현 기준으로 쓰이는 문서가 되도록 하겠습니다.

### 6-4. 범위를 늘리기보다, 기준을 지키는 쪽을 우선하기
구현 가능하다고 느껴져도,  
프로젝트 핵심이 흐려질 정도의 확장은 최대한 늦추거나 제외하겠습니다.

## 7. 우리가 하지 않으려는 방향

이 부분은 분명하게 적어두고 시작하는 것이 좋겠습니다.

### 7-1. AI에 전적으로 의존하는 바이브 코딩
이번 프로젝트에서 AI는 보조 도구로만 활용하겠습니다.  
구현 자체를 AI에게 맡기고, 팀은 결과만 붙이는 방식은 지양합니다.

### 7-2. 빠르게 결과만 뽑아내는 방식
겉으로 돌아가는 화면만 만들고 구조가 정리되지 않은 상태로 끝내는 방향은 지양하겠습니다.  
이번 프로젝트의 목적은 “빨리 하나 만들기”가 아니라,  
**왜 이런 구조를 택했고 어떻게 협업했는지 남기는 것**입니다.

### 7-3. 무리한 기술 확장
현재 팀이 감당하기 어려운 기술을 한 번에 얹는 방식은 피하겠습니다.  
새로운 기술 자체보다, 지금 수준에서 확실히 이해할 수 있는 범위를 지키는 것이 더 중요합니다.

### 7-4. 백엔드, 인증, 상태 관리, 데이터 저장, 예외 처리, 계층 구조를 한 번에 다 하려는 시도
이런 방식은 프로젝트 경험이 적은 팀에게 과부하가 될 가능성이 큽니다.  
우리는 필요한 흐름만 남기고, 나머지는 과감히 덜어내는 쪽을 선택하겠습니다.

### 7-5. DB까지 포함한 다층 구조를 무리하게 도입하는 것
이번 프로젝트에서는 DB를 필수 범위로 두지 않겠습니다.  
JSON 기반 데이터 처리만으로도 충분히 설계와 흐름 경험을 만들 수 있다고 봅니다.

### 7-6. Spring을 프로젝트 핵심 성과처럼 과장하는 것
Spring을 붙이게 되더라도, 그것을 프로젝트의 대표 성과처럼 말하지 않겠습니다.  
핵심은 프레임워크 이름이 아니라, 팀이 직접 설계하고 구현하고 이해한 범위입니다.

### 7-7. 여러 프로젝트를 병행하며 범위를 넓히는 것
이번 스터디는 결과물 1개를 기준으로 운영하겠습니다.  
프로젝트 수를 늘리는 것보다, 하나를 끝까지 가져가는 경험이 훨씬 중요합니다.

## 8. AI 사용 원칙

AI는 아예 금지할 필요는 없다고 생각합니다.  
다만 사용 원칙은 분명해야 합니다.

### 8-1. AI는 보조 도구로만 사용
가능한 사용 범위는 아래 정도로 제한합니다.

- 개념 정리
- 구조 검토
- 이해가 부족한 부분에 대한 설명 보조
- 예외 케이스 점검
- 문서 표현 다듬기

### 8-2. 구현 자체를 AI에 맡기는 방식은 지양
팀이 직접 이해하지 못한 코드,  
설명할 수 없는 구조,  
왜 그렇게 동작하는지 모른 채 붙여 넣은 구현은 이번 프로젝트의 핵심 성과로 보지 않겠습니다.

### 8-3. AI를 사용했다면 팀이 공유 가능해야 함
AI 도움을 받은 경우에는  
어떤 프롬프트를 사용했고, 어떤 부분을 참고했고, 최종적으로 무엇을 이해했는지 팀 내에서 공유하는 것을 원칙으로 하겠습니다.

즉, AI는 개인의 지름길이 아니라  
팀이 함께 검토 가능한 보조 수단이어야 합니다.

## 9. 결과물을 평가하는 기준

이번 프로젝트는 단순히 “돌아간다 / 안 돌아간다”로만 평가하지 않겠습니다.  
아래 기준을 함께 보겠습니다.

- 기능이 분명하게 나뉘어 있는가
- 역할 분담이 실제로 작동했는가
- Git 협업 흐름이 유지되었는가
- README와 문서가 남아 있는가
- 구조 설계 이유를 설명할 수 있는가
- 예외 메시지와 이동 흐름이 정리되어 있는가
- 범위를 통제하면서도 결과물 1개를 끝까지 완성했는가

즉, 결과물의 화려함보다  
**협업 구조, 설계 이유, 작업 방식, 문서화, 시행착오의 정리**를 더 중요한 성과로 보겠습니다.

## 10. 마무리 제안

정리하면, 이번 프로젝트는 풀스택 서비스를 완성하는 것이 목표가 아닙니다.  
백엔드나 DB를 억지로 붙여서 기술적으로 커 보이는 결과물을 만드는 것도 목표가 아닙니다.

우리가 이번에 해보고 싶은 것은 더 작지만 더 분명합니다.

- 협업 흐름을 실제로 경험해보기
- 구조를 먼저 고민한 뒤 구현해보기
- 역할을 나누고 책임을 분명히 해보기
- 제한된 범위의 결과물 하나를 끝까지 완성해보기
- 그 과정을 README와 문서로 남겨보기

프로젝트 주제와 기술 선택은 이후 회의에서 함께 정하면 되지만,  
적어도 운영 방향만큼은 이 기준 위에서 시작했으면 합니다.

이번 프로젝트가 “대단한 서비스 하나”로 남지 않아도 괜찮습니다.  
대신 나중에 돌아봤을 때  
“우리가 어떤 기준으로 협업했고, 왜 그렇게 설계했고, 어디까지 구현했고, 무엇을 배웠는지”가 분명히 남는 프로젝트가 되었으면 합니다.
