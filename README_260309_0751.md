# GitHub Repository 기반 프로젝트 협업 스터디 모집

## 스터디 모집 배경
저는 관통 프로젝트 외에는 팀 프로젝트 경험이 거의 없고, 스터디를 운영해본 경험도 없습니다.  
그래서 진행이 완벽하게 매끄럽지는 않을 수 있습니다.  

다만 지금까지 배운 내용을 바탕으로 직접 협업해보고, GitHub repository를 활용한 프로젝트 경험을 함께 쌓아보고 싶어 이 스터디를 구상하게 되었습니다.  
함께 부딪히며 경험을 쌓고 끝까지 결과물을 만들어보고 싶은 분과 함께하고 싶습니다.

---

## 1. 스터디 소개
이 스터디는 **GitHub repository를 활용한 협업 경험**을 중심으로,  
약 **2개월 동안 하나의 프로젝트를 함께 완성해보는 프로젝트형 스터디**입니다.

협업 연습이 핵심이지만, **최소 1개의 결과물 완성**을 목표로 합니다.

---

## 2. 스터디 목표
### 협업 목표
- GitHub repository를 활용한 협업 흐름 경험
- 역할 분담 및 작업 기록 경험
- README 작성 및 결과물 정리 경험

### 결과물 목표
- 약 2개월 동안 **프로젝트 1개 완성** (진행 상황에 따라 최대 2개)
- 팀 상황과 일정에 따라 결과물을 포트폴리오로 활용 가능하도록 정리

### 학습 목표
- 문제 해결을 중심으로 주제를 선정하고 구현
- 초기 설계 단계에서 구조를 고민하고, 이를 바탕으로 구현 진행
- 프로젝트 유형에 따라 **Java 또는 JS**를 실제로 적용해보는 경험 축적
- 필요 시 AI를 활용해 구조를 검토하되, 구현 자체는 최대한 직접 진행

---

## 3. 진행 방식
- 기본 진행 : **온라인**
- 실시간 회의 및 협업 : **Discord**
- 연락 및 소통 : **카카오톡**
- 필요 시 SSAFY 내에서 **오프라인 미팅** 가능
- 모집 완료 후 바로 아이디어 구상 시작 예정
- 다음 주 월요일 시험 일정이 있으므로,  
초반에는 구현보다 기획/정리 및 Git 활용부터 진행할 수 있습니다.

---

## 4. 프로젝트 방향
- 기본 프로젝트 수는 **1개**
- 예상 진행 기간은 **약 2개월**
- 프로젝트 주제는 **구성원 확정 후 함께 논의하여 결정**
- 공모전 일정이나 주제 적합성이 맞는 경우, 제출 가능성도 검토
- 1차 프로젝트의 핵심은 **다양한 기술 스택보다 GitHub 협업 경험**
- 기술 범위는 무리하게 넓히지 않고, 현재까지 배운 내용을 중심으로 설정

---

## 5. 기술 범위
### 기본 기준
- 프론트는 Front(Web) 수업의 **HTML / CSS / JS** 기준
- 백엔드는 필수 **X**
- DB는 현재 필수 범위로 두지 않음
- 데이터 저장/관리는 **JSON 기반**으로 고려 중

### 언어 선택 방향
- 웹 서비스형 프로젝트라면 **JS** 활용 가능성이 높습니다.
- 웹이 아닌 구조라면 **Java** 활용 가능성이 높습니다.
- 두 개의 프로젝트를 진행할 경우 각각 다른 언어를 선택합니다.

### 확장 가능성
- 일정과 팀 역량이 허용될 경우 간단한 백엔드 또는 데이터 관리 요소를 추가 검토
- <s>시간적 여유가 충분할 경우에만 MySQL 등 DB 활용 검토</s>

---

## 6. 협업 방식에서 최소 목표로 두는 것
이번 스터디에서는 아래 항목들을 최소 목표로 두고 진행하려고 합니다.

- **branch 분리**
- **commit 가독성**
- **README 정리**

단순히 저장소만 만드는 것이 아니라, **작업 분담/기록/결과물 문서화**까지 경험하는 것을 목표로 합니다.

---

## 7. 구조 설계 방향
이 스터디는 단순 구현만 하는 것이 아니라,  
초기 설계 단계에서 **구조를 먼저 정리하고 그에 맞춰 구현을 진행하는 경험**도 목표로 합니다.

현재 시스템 설계나 패키징 경험이 많지 않다는 점을 고려하여, 다음과 같은 방향으로 접근하려고 합니다.

1. 역할 분담
2. 역할 기반 패키지 구조 설계
3. 패키지/모듈 분리 시도
4. (선택 사항) AI를 활용한 구조 검토

---

## 8. AI 사용
- 개인의 AI 사용을 제한하지는 않습니다.
- 다만 **코드 AI가 직접 작성하게 하는 방식은 지양**합니다.
- 개념 정리, 구조 검토, 미숙한 부분의 이해 보조 등은 활용 가능합니다.
- 팀에서 직접 구현하기 어려운 부분에 AI 도움을 받은 경우,  
  **사용한 프롬프트를 텍스트 파일로 정리하여 repository에 공유**하려고 합니다.

즉, 개인 학습 또는 팀 토의를 통해 문제 해결을 시도하되,  
AI를 활용하는 경우 프롬프트를 공유하고, **해당 내용을 직접 이해하는 것**을 제안하고 싶습니다.

- 현재 개발 실력은 크게 중요하지 않습니다.
- Git / Java / 알고리즘 / Front(Web) / DB 활용 능력이 부족해도 괜찮습니다.
- 다만 아래 두 가지는 중요하게 생각합니다.

### 중요하게 보는 기준
1. **실제 참여 가능한 시간 확보**
2. **끝까지 성실하게 참여할 의지**

- 평일 최소 **1시간**
- 주말 최소 **3시간**

위 시간은 개인 학습 시간을 제외한 **실제 스터디 참여 가능 시간**을 기준입니다.
